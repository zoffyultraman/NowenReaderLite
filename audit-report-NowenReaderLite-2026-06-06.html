📝 总体评估

📋 总体评估

项目概况：NowenReaderLite 是一款 iOS 漫画/小说阅读器，支持 AI 超分辨率（CoreML tile 推理）、自托管服务器连接、阅读统计等功能。Swift/SwiftUI 技术栈，36 个 Swift 文件共 7,220 行代码。架构为 Feature-based 分层（App/Core/Features/Models）。

主要风险：① 测试覆盖率为零——没有任何测试文件、测试目标或 CI/CD 流水线，所有变更完全依赖人工验证；② ImageUpscaler 在处理大图时可能分配 1GB+ 内存导致 OOM 崩溃；③ ATS 完全禁用且允许 HTTP 连接，凭据可被中间人截获；④ APIClient 是全局 singleton 且无依赖注入，所有 ViewModel 直接耦合，无法进行单元测试。

亮点：① Keychain 存储密码（kSecAttrAccessibleWhenUnlockedThisDeviceOnly）；② tile 推理架构设计合理（固定 stride + center crop）；③ 日志统一通过 AppLogger；④ 图片缓存使用 SHA-256 哈希防止路径遍历；⑤ URL 验证逻辑完整（拒绝嵌入凭据、校验 scheme/host）。

优先修复：① 添加基础单元测试（至少覆盖 APIClient、ImageUpscaler、PaginationService）；② 修复 ImageUpscaler OOM 风险（增量写入输出缓冲区）；③ 为 APIClient 引入协议抽象和依赖注入；④ 拆分超大文件（ComicReaderView 727 行、NovelReaderView 779 行）。

总评：C（4.6 分）——项目功能基本可用，但工程基础设施严重缺失。零测试覆盖 + 零 CI/CD 意味着每次发布都是赌博。代码质量中等，存在明显的架构债务（singleton 耦合、god object、DRY 违反）。在投入新功能之前，应优先建立测试基础设施和修复高风险稳定性问题。

🔍 详细发现

1. ImageUpscaler 大图输出缓冲区 OOM 崩溃

High Confirmed
📁 稳定性 📍 ImageUpscaler.swift:185 🎯 tileInference
问题：输出像素缓冲区按完整 4x 放大尺寸分配。对于 4096×4096 输入（maxDimension 上限），输出为 16384×16384，分配 16384×16384×4 = 1,073,741,824 字节（1GB）。在 4GB RAM 设备上将触发 OOM kill。
ImageUpscaler.swift:185 var finalPixels = [UInt8](repeating: 0, count: finalWidth * finalHeight * 4) // finalWidth = inputWidth * 4, finalHeight = inputHeight * 4 // 4096 * 4 = 16384 → 16384 * 16384 * 4 = 1GB
最小修复
// 方案 A：降低 maxDimension 上限（如 2048） // 方案 B：改为逐 tile 写入 CGContext，不预分配全量缓冲区 // 方案 C：使用 CGContext 直接绘制 tile，避免中间像素数组
⏱ 预估工时： 2-4 小时  ·  回归测试： 用 3000px+ 图片测试 upscale，监控内存峰值

2. ATS 完全禁用 + HTTP 凭据明文传输

High Confirmed
📁 安全性 📍 Info.plist:27-28, APIClient.swift:33-49 🎯 setServerURL / Info.plist
问题：NSAllowsArbitraryLoads=true 完全禁用 ATS，允许所有 HTTP 明文连接。setServerURL 同时接受 http:// 和 https://。登录注册通过 JSON POST 发送用户名密码，HTTP 下凭据以明文传输，可被网络嗅探或中间人攻击截获。
Info.plist: <key>NSAllowsArbitraryLoads</key> <true/> APIClient.swift:37-38: guard let parsed = URL(string: trimmed), parsed.scheme == "http" || parsed.scheme == "https"
最小修复
// 1. 移除 NSAllowsArbitraryLoads，改用 NSExceptionDomains // 2. setServerURL 中对 HTTP 连接显示安全警告 // 3. 登录/注册前检查 scheme，HTTP 下弹窗提醒用户
⏱ 预估工时： 1-2 小时  ·  回归测试： 验证 HTTPS 连接正常，HTTP 下显示警告

4. APIClient singleton 全局耦合，无法注入测试

High Confirmed
📁 设计 📍 所有 ViewModel（7 处） 🎯 APIClient.shared
问题：所有 ViewModel 直接持有 APIClient.shared（ComicReaderViewModel、NovelReaderViewModel、LibraryViewModel、StatsViewModel、ContinueReadingViewModel、FavoritesViewModel、DetailViewModel）。无协议抽象、无初始化注入，使得所有 ViewModel 无法在不访问真实网络的情况下进行单元测试。同时 6 个 View 通过 @ObservedObject 监听 singleton，任意 @Published 属性变化都会触发全部 6 个 View 重绘。
ComicReaderViewModel.swift:15: private let api = APIClient.shared NovelReaderView.swift:548: private let api = APIClient.shared LibraryView.swift:139: private let api = APIClient.shared StatsView.swift:443: private let api = APIClient.shared HomeView.swift:447: private let api = APIClient.shared FavoritesView.swift:67: private let api = APIClient.shared ComicDetailView.swift:251: private let api = APIClient.shared
最小修复
// 1. 定义 protocol APIServiceProtocol // 2. APIClient 遵循该协议 // 3. ViewModel init 接受 APIServiceProtocol 参数 // 4. Preview 和测试中注入 mock 实现
⏱ 预估工时： 1 天  ·  回归测试： 所有 ViewModel 可独立单元测试

5. TaskGroup.next()! 强制解包可崩溃

High Confirmed
📁 稳定性 📍 ServerListView.swift:231 🎯 服务器切换逻辑
问题：TaskGroup.next() 返回 Optional，强制解包在 group 被提前取消时会崩溃。快速点击服务器切换按钮可能触发此路径。
ServerListView.swift:231: let first = await group.next()!
最小修复
guard let first = await group.next() else { return .timeout }
⏱ 预估工时： 5 分钟  ·  回归测试： 快速连续点击服务器切换按钮

6. NovelReaderView .onChange 链含 500ms sleep 延时 hack

High Confirmed
📁 前端状态 📍 NovelReaderView.swift:87-100 🎯 章节切换恢复逻辑
问题：三个 .onChange 处理器级联触发：pages.count 变化 → restorePosition()；currentChapter 变化 → Task.sleep(500ms) → restorePosition()；fontSize 变化 → UserDefaults 写入 + repaginate()。500ms sleep 是脆弱的时序 hack，页面加载快慢不同时行为不确定。
.onChange(of: viewModel.currentChapter) { _, _ in Task { try? await Task.sleep(nanoseconds: 500_000_000) // fragile! restorePosition() } }
最小修复
// 用 Combine publisher 的 .debounce 或 .removeDuplicates 替代 // 或用单一状态机统一处理章节/页面/字体变化
⏱ 预估工时： 2-4 小时  ·  回归测试： 快速连续翻章，验证位置恢复正确

7. View 层直接调用 APIClient 绕过 ViewModel

High Confirmed

问题：SwiftUI View 和 UIKit Controller 直接调用 APIClient.shared 进行网络请求（fetchPages、pageImageURL、authenticatedRequest）。网络层与 UI 层零隔离，无法 mock 测试。

最小修复
// 通过 ViewModel 或 Representable 的闭包传递 URL 和 Request // UIKit 层不应直接访问网络 singleton
⏱ 预估工时： 4-8 小时  ·  回归测试： 漫画阅读器图片加载、翻页、预加载功能回归
